# מהו RAG?

RAG היא טכניקה בבינה מלאכותית המשלבת שליפת מידע עם יצירת טקסט. במקום לסמוך רק על הידע שנלמד במודל, RAG מאפשר למערכת לחפש מידע רלוונטי ממאגר מסמכים ולהשתמש בו כדי לייצר תשובות מדויקות ועדכניות יותר.

## למה RAG חשוב?

RAG פותר כמה בעיות מרכזיות:

- **המצאת עובדות** - מודלים גדולים לפעמים ממציאים עובדות. RAG מבסס את התשובה על מסמכים אמיתיים.
- **מידע לא עדכני** - מודלים נוצרים בנקודת זמן מסוימת. RAG מאפשר להוסיף מידע חדש בקלות.
- **ידע ספציפי** - אפשר להוסיף מסמכים פנימיים של החברה או תחום מומחיות.

## אדריכלות RAG - שלבי המערכת

מערכת RAG קלאסית מורכבת מהשלבים הבאים:

1. **טעינה** - קריאת מסמכים מקובץ או מסד נתונים
2. **חלוקה** - פירוק המסמכים לנתחים קטנים יותר
3. **הטמעה** - המרה של הטקסט לוקטורים מספריים
4. **אחסון** - שמירת הוקטורים במאגר וקטורי
5. **שליפה** - חיפוש הנתחים הרלוונטיים ביותר לשאילתה
6. **סינתזה** - שילוב הנתחים וייצור תשובה מקיפה

## ההבדל בין הטמעה לוקטוריזציה

**הטמעה (Embedding)** היא תהליך המרה של טקסט לוקטור - רצף מספרים המייצג את המשמעות הסמנטית של הטקסט. שני משפטים עם משמעות דומה יהיו קרובים במרחב הוקטורי.

**וקטוריזציה (Vectorization)** היא מושג רחב יותר המתייחס להמרה של כל סוג נתון למספרים. ההבדל המרכזי הוא שהטמעה מייצגת **משמעות** בעוד שוקטוריזציה פשוטה מייצגת **תדירות מילים** בלבד.

## טכנולוגיות נפוצות

- **LlamaIndex ו-LangChain** - ספריות לבניית מערכות RAG
- **Pinecone, ChromaDB ו-Weaviate** - מאגרי וקטורים
- **OpenAI ו-Cohere** - שירותי הטמעה ומודלי שפה
- **FAISS** - ספריית חיפוש וקטורים מהירה לשימוש מקומי

## דוגמאות שימוש

- מערכות תמיכה לקוחות המבוססות על מדריכי מוצר
- עוזרים אישיים שיודעים על מסמכים פנימיים של חברה
- מערכות שאלות ותשובות על קוד או תיעוד טכני
- בוטים שמשלבים ידע עדכני מאינטרנט

## יתרונות מול חסרונות

**יתרונות:**

- שליטה במקורות המידע
- אפשרות לעדכן מידע בקלות
- מקור לתשובה ניתן לאימות
- חיסכון בעלויות אימון מחדש של מודלים

**חסרונות:**

- צורך בתשתית נוספת (vector database)
- עלות של embedding וחיפוש
- איכות התשובה תלויה באיכות המסמכים
- מורכבות טכנית גבוהה יותר

## RAG מול Fine-Tuning - מה ההבדל?

### Fine-Tuning

**Fine-Tuning** הוא תהליך בו לוקחים מודל קיים ומאמנים אותו מחדש על נתונים ספציפיים. זה משנה את המשקולות של המודל עצמו.

**מתי להשתמש:**

- כשצריך שהמודל "ידע" מידע בעל פה
- כשצריך סגנון כתיבה ספציפי
- כשהמידע לא משתנה הרבה
- כשצריך ביצועים מהירים מאוד

**חסרונות:**

- יקר (צריך GPU, זמן, מומחיות)
- קשה לעדכן (צריך אימון מחדש)
- סיכון ל-overfitting
- המודל יכול עדיין להמציא עובדות

### RAG

**RAG** לא משנה את המודל - הוא מחפש מידע רלוונטי ומזין אותו למודל בזמן אמת.

**מתי להשתמש:**

- כשהמידע משתנה לעתים קרובות
- כשצריך מקורות מאומתים
- כשיש מסמכים ארוכים
- כשהתקציב מוגבל

**יתרונות:**

- קל לעדכן (פשוט מוסיפים מסמכים)
- שקוף (רואים מאיפה הגיעה התשובה)
- זול יותר מאימון מחדש
- מדויק יותר למידע עדכני

### האופציה השלישית: Fine-Tuning + RAG

במקרים מסוימים, משלבים את שני הגישות:

- Fine-Tuning לסגנון ולמבנה תשובות
- RAG למידע עובדתי עדכני

זה הפתרון הטוב ביותר אבל גם היקר ביותר.

## מקרי שימוש בעולם האמיתי

### 1. תמיכת לקוחות חכמה

חברה יכולה להזין את כל מדריכי המוצר, FAQ, וטיקטים קודמים למערכת RAG. כשלקוח שואל שאלה, המערכת:

- מחפשת במדריכים את המידע הרלוונטי
- מחפשת טיקטים דומים שנפתרו בעבר
- מייצרת תשובה מותאמת אישית

**דוגמה:**

```
לקוח: "האפליקציה קורסת כשאני מנסה לייצא קובץ CSV"
RAG מוצא:
- מדריך: "ייצוא CSV תומך רק בקבצים עד 10MB"
- טיקט קודם: "בעיה דומה נפתרה על ידי עדכון לגרסה 2.5"
תשובה: "נראה שהבעיה יכולה להיות קשורה לגודל הקובץ (מקסימום 10MB).
אם הקובץ קטן יותר, נסה לעדכן לגרסה 2.5 שמתקנת באג דומה."
```

### 2. עוזר למפתחים (Code Assistant)

RAG יכול לעזור למפתחים על ידי חיפוש בתיעוד, דוגמאות קוד, ושיחות ב-Slack.

**דוגמה:**

```
מפתח: "איך אני משתמש ב-API החדש לשליחת מיילים?"
RAG מוצא:
- תיעוד API עם דוגמת קוד
- דיון ב-Slack שבו מישהו פתר בעיה דומה
תשובה: "הנה דוגמה לשימוש ב-API (+ קוד). שים לב שצריך להוסיף
API key בheaders - כמו שעשה דני בפרויקט האחרון."
```

### 3. מערכת משפטית/רפואית

תחומים שבהם דיוק הוא קריטי ויש המון מסמכים.

**דוגמה רפואית:**

```
רופא: "מה התופעות הלוואי של תרופה X עם תרופה Y?"
RAG מחפש ב:
- עלוני התרופות
- מאמרים מדעיים
- הנחיות משרד הבריאות
תשובה: "לפי עלון התרופה מ-2025, השילוב עלול לגרום ל...
מקור: [קישור למסמך המקורי]"
```

### 4. חיפוש בארכיון חברה

חברות צוברות שנים של מסמכים - פרוטוקולים, דוחות, החלטות.

**דוגמה:**

```
מנכ"ל: "למה החלטנו לא להיכנס לשוק הסיני ב-2023?"
RAG מחפש:
- פרוטוקולים של ישיבות דירקטוריון
- דוחות שוק
- מצגות אסטרטגיה
תשובה: "לפי פרוטוקול מרץ 2023, הוחלט לדחות בגלל: 1) רגולציה
מורכבת 2) עלות גבוהה של התאמה מקומית. ראה מצגת מצורפת."
```

## Best Practices ליישום RAG

### 1. איחוד המסמכים (Chunking)

**הבעיה:** מסמך גדול מדי או קטן מדי לא עובד טוב.

**פתרון:**

- גודל מומלץ: 256-512 tokens לchunk
- overlap: 10-20% בין chunks
- שמור на קונטקסט - אל תחתוך באמצע משפט!

**דוגמה רעה:**

```
Chunk 1: "...והחברה החליטה"
Chunk 2: "לפתוח סניף בלונדון בגלל..."
```

**דוגמה טובה:**

```
Chunk 1: "...והחברה החליטה לפתוח סניף בלונדון"
Chunk 2: "לפתוח סניף בלונדון בגלל הביקוש הגבוה באירופה"
```

### 2. בחירת מודל Embeddings

מודלים שונים טובים לדברים שונים:

- **OpenAI text-embedding-ada-002**: כללי, טוב לאנגלית
- **Cohere embed-multilingual**: מעולה לעברית ושפות נוספות
- **BGE models**: חינמי, טוב לשימוש מקומי

**טיפ:** אם יש לך עברית במסמכים - חובה multilingual!

### 3. כמה Chunks להחזיר?

**Top-K** זה כמה תוצאות מחזירים מהחיפוש.

- יותר מדי (K=10): אינפורמציה לא רלוונטית, תשובה מבולבלת
- מעט מדי (K=1): חסר קונטקסט, תשובה חלקית
- מומלץ: K=3-5

### 4. Metadata חשובה!

הוסף metadata לכל chunk:

```python
{
    "text": "התוכן של הטקסט...",
    "source": "report_2024.pdf",
    "date": "2024-05-15",
    "author": "דני כהן",
    "category": "פיננסי"
}
```

אז תוכל לסנן: "תן לי רק מידע מ-2024" או "רק דוחות פיננסיים"

### 5. Prompt Engineering

התבנית שאתה נותן ל-LLM חשובה מאוד!

**Prompt גרוע:**

```
Context: {docs}
Question: {query}
```

**Prompt טוב:**

```
אתה עוזר מועיל. ענה על בסיס המסמכים הבאים בלבד.
אם התשובה לא נמצאת במסמכים, אמר "אין לי מידע על זה".

מסמכים:
{docs}

שאלה: {query}

תשובה (בעברית, קצרה וברורה):
```

## אתגרים נפוצים ופתרונות

### אתגר 1: "המערכת מחזירה תוצאות לא רלוונטיות"

**סיבות אפשריות:**

- מודל Embeddings לא מתאים לשפה
- Chunks גדולים מדי או קטנים מדי
- Query לא ברורה

**פתרון:**

- שנה למודל multilingual
- התאם את גודל ה-chunks
- הוסף Query Expansion (הרחב את השאלה)

**דוגמה:**

```
Query מקורי: "מנכ"ל"
Query מורחב: "מנכ"ל CEO מנהל כללי"
```

### אתגר 2: "התשובות ארוכות מדי"

**פתרון:**

- שנה את response_mode ל-"compact" במקום "tree_summary"
- הוסף להוראה: "ענה בקצרה, עד 3 משפטים"
- הגדל temperature ל-0.3 (פחות יצירתי = יותר תמציתי)

### אתגר 3: "המערכת ממציאה עובדות"

**פתרון:**

- הוסף להוראה: "אם לא יודע - תגיד 'לא יודע'"
- הקטן temperature ל-0.1
- הוסף בדיקת עקביות: בקש מהמודל לצטט מהמסמך

### אתגר 4: "איטי מדי!"

**פתרונות:**

- השתמש ב-Caching (שמור embeddings של queries פופולריים)
- העבר ל-Vector DB מהיר יותר (Pinecone/Qdrant במקום FAISS)
- הקטן את K (פחות chunks)
- שקול Async processing

## מתי RAG לא מתאים?

למרות היתרונות, RAG לא תמיד הפתרון הטוב ביותר:

### ❌ אל תשתמש ב-RAG כש:

1. **המידע קצר ופשוט**
   - אם יש לך 10 שאלות FAQ פשוטות, מספיק dictionary רגיל
   - RAG זה overkill

2. **צריך תשובות מיידיות מאוד מאוד**
   - RAG לוקח ~1-3 שניות (embed + search + generate)
   - אם צריך <100ms, RAG לא מתאים

3. **אין מסמכים מובנים**
   - אם המידע מפוזר, לא מתועד, או בראשים של אנשים
   - קודם כל תתעד!

4. **צריך יצירתיות מלאה**
   - אם אתה רוצה שה-LLM יהיה יצירתי (שירים, סיפורים)
   - RAG מגביל אותו למסמכים

### ✅ השתמש ב-RAG כש:

- יש לך המון מסמכים (>50 עמודים)
- המידע משתנה לעתים קרובות
- צריך מקורות למידע
- חשוב שהתשובה תהיה מדויקת ועדכנית
- יש תקציב מוגבל (בהשוואה ל-Fine-Tuning)

## טיפים לשיפור ביצועים

### 1. Hybrid Search

במקום רק Semantic Search, שלב גם Keyword Search:

- **Semantic**: מוצא משמעות (RAG ≈ Retrieval-Augmented Generation)
- **Keyword**: מוצא מילים מדויקות (API, שמות, תאריכים)

**דוגמה:**

```
Query: "באיזה תאריך שוחרר GPT-4?"
Semantic לבד: עלול להחזיר תוצאות על GPT-3, GPT-3.5
Hybrid: ימצא בדיוק GPT-4 בזכות keyword search
```

### 2. Re-Ranking

לאחר שקיבלת K=10 תוצאות, העבר אותן דרך מודל נוסף ש"מדרג מחדש".

```
Initial Retrieval (fast, K=10):
[doc1, doc5, doc3, doc7, doc2, doc9, doc4, doc6, doc8, doc10]

Re-Ranking (slow but accurate, top 3):
[doc5, doc1, doc7]  ← רק אלה עוברים ל-LLM
```

### 3. Query Expansion

הרחב את השאלה לפני החיפוש:

```
Original: "מה זה RAG?"
Expanded: "מה זה RAG? Retrieval Augmented Generation שליפת מידע"
```

### 4. Contextual Compression

לאחר שמצאת chunks רלוונטיים, "דחוס" אותם להוציא רק את המשפטים הרלוונטיים.

```
Original Chunk (200 words):
"החברה נוסדה ב-1995... [lots of history]...
בשנת 2024 השיקה מוצר X שכולל RAG..."

Compressed (30 words):
"בשנת 2024 השיקה מוצר X שכולל RAG..."
```

זה חוסך tokens ומשפר איכות!

## סיכום - המפתח להצלחה עם RAG

1. ✅ **מסמכים איכותיים** - Garbage in, Garbage out
2. ✅ **Chunks נכונים** - לא גדול מדי, לא קטן מדי
3. ✅ **Embeddings מתאימים** - multilingual לעברית!
4. ✅ **Prompt טוב** - הנחיות ברורות ל-LLM
5. ✅ **K מאוזן** - בדרך כלל 3-5 תוצאות
6. ✅ **Metadata** - קטגוריות, תאריכים, מקורות
7. ✅ **טסטים** - בדוק על שאלות אמיתיות מהמשתמשים
8. ✅ **מוניטורינג** - עקוב אחרי איכות התשובות

## לסיכום - RAG זה המהפכה

RAG משנה את הכללים של AI:

- מודלי שפה גדולים + מידע עדכני
- תשובות מדויקות + מקורות מאומתים
- עלות נמוכה + גמישות עדכון

זו הסיבה שכמעט כל אפליקציית AI חדשה משתמשת ב-RAG!
